# qa-swarm 변경 대장 (메타-PLM)

> **한 줄 요지:** 이 대장은 qa-swarm 하네스가 자기 자신에게 가한 변경을 하나도 빠짐없이 기록하는 장부다. 하네스의 버전을 올리려면 반드시 아래 정해진 형식으로 이 대장에 먼저 기록해야 하며, 기록 없이 릴리스된 하네스를 직접 고치는 것은 금지한다.

이 하네스는 자기 자신을 **PLM(Product Lifecycle Management, 제품 수명주기 관리 — 제품의 변경을 이력으로 관리하는 방식)처럼** 다룬다. 그래서 이름이 "메타-PLM 변경 대장"이다. 규칙은 다음과 같다.

- **버전 올리기(bump)는 반드시 아래 형식의 변경 기록을 남긴 경우에만 한다.** 이미 배포(릴리스)된 하네스를 기록 없이 직접 수정하는 것은 금지한다.
- **개선은 창고(Template-repository)에서만 한다.** 창고로 올리는 변경(상향)은 PR(Pull Request, 변경 제안·검토 요청)로 처리하고, 창고에서 각 저장소로 내려가는 배포(하향)는 skill-sync 동기화 봇이 자동으로 수행한다.

이 대장의 무게(엄격함)는 변경의 **가역성(되돌릴 수 있는 정도)에 비례한다.** 여기서는 DRBFM(Design Review Based on Failure Mode, 고장 모드 기반 설계 검토 — 변경이 일으킬 수 있는 고장을 미리 따져 보는 기법)의 **사고방식(변경을 추적하고 그 영향을 분석하는 것)만** 가져오고, DRBFM의 **무거운 절차(승인 라우팅 등)는 가져오지 않는다.** 즉 파일 하나를 고치는 데 결재 과정을 걸지는 않는다.

## 기록 형식

새 버전을 기록할 때는 아래 형식을 그대로 따른다.

```
## vX.Y.Z — <제목> (일자)
- As-Is:      변경 전 상태
- To-Be:      변경 후 상태
- 문제:        무엇이 문제였나
- 개선 이유:   왜 이 변경이 더 나은가
- 역기능(예측): 이 변경이 새로 만들 수 있는 부작용 / 고장 모드 (변경 시점의 예측)
- 역기능(실측): 이후 검증 사이클(loop)에서 실제로 관측된 부작용 — 예측과 대조해 갱신
- 영향 범위:   바뀐 파일 / 구성물
```

## v0.3.0 — 분모 완료 게이트 (2026-07-16)

- **As-Is:** 준비 상태 보드에 "코드에서 분모 생성" 항목은 있었으나, 그것을 실제로 생성했는지 강제하지 않았고, 분모 없이도 L1을 "완료"로 선언하는 것을 막는 규칙이 없었다.
- **To-Be:** "코드에서 분모 생성"을 **완료 게이트**로 격상했다. 이 항목이 🟢/🟡로 채워지기 전에는 L1 커버리지를 "충분/완료"로 선언할 수 없고, 분모가 없으면 L1은 "측정 불가(분모 미생성)"로 보고한다. SKILL.md 보고 절과 qa-contract.md 준비 상태 보드 양쪽에 명시했다.
- **문제:** 첫 실전(loop-1/2)에서 이 스킬을 돌린 실행자가 1단계(분모 생성)를 건너뛰고 이미 쓰여 있던 대표 여정 7개(user-flows J1~J7)만 페르소나로 돌린 뒤 "L1 마무리"로 오판했다. 분모가 없어 "얼마나 봤나"가 측정된 적이 없는데도 완료로 착각한 것 — 이 스킬이 막으려던 "눈에 띄는 머리만 보고 꼬리를 빠뜨림"을 스킬 스스로 허용한 셈이다. 실측 결과 동작 48개 중 8개(약 17%)만 실제 검증됐고 나머지는 미지였다.
- **개선 이유:** 완전성 보증이 실행자의 성실함에만 의존하면 이미 안전장치가 아니다. 완료 선언을 분모 존재에 묶어 두면, 실행자가 성급해도 구조가 "측정 불가"로 멈춰 세운다.
- **역기능(예측):** 게이트가 형식적 분모(라우트만 열거하고 상태·경계는 비운 저품질 분모)로 우회될 수 있다. 완화: 분모 등급(스크립트=🟢 / 판단=🟡 저신뢰)을 원장에 함께 표기해, 저품질 분모 위의 커버리지는 저신뢰로 드러낸다.
- **역기능(실측):** _TBD — 다음 적용 후._
- **영향 범위:** `SKILL.md`(보고 절 완료 게이트 bullet, 상태 라벨), `qa-contract.md`(준비 상태 보드 표 아래 게이트 설명).

## v0.2.0 — 루프 변경점 보고(As Is/To Be) 규약 + loop-1/2 실측 반영 (2026-07-16)

- **As-Is:** 보고 산출물이 채팅 5종 요약(발견·원장·핸드오프·결정화·사람검증 후보)뿐이었다. 이것은 흘러가 버려서, "고친 것의 전후를 시각적으로 누적·보존하는" 산출물이 없었다. 또 첫 실전(loop-1/2)에서 얻은 실측 교훈(특히 "판단 시점")이 문서에 없었다.
- **To-Be:** 두 가지를 넣었다. (1) **루프 한 턴(고침 → 재검증)이 끝날 때마다 As Is / To Be 변경점 보고서를 한 링크로 누적 갱신**하는 규약을 `SKILL.md` 보고 절에 추가했다. 손본 항목마다 실제 화면 캡처(수정 전/재검증 후), "변경점" 한 줄, 재검증 결과를 담고, KimPM 깊은 보고 방식(소제목만 훑어도 흐름이 섬)을 따르며, 자체완결 HTML(캡처를 data-URI로 인라인)로 낸다. (2) **판단 시점 규칙**을 명문화했다 — "저장 직후·새로고침 전 화면을 결과로 판단하지 말 것".
- **문제:** 검증 결과가 축적·공유되지 않아 이해관계자가 진척을 한눈에 볼 수 없었고, 개별 검증원이 새로고침 전 화면을 결과로 오판해 거짓 경보를 냈다(loop-2에서 "영향 부품 전면 고장"·"죽은 링크" 두 건 발생, 오케스트레이터 재현으로 걸러냄).
- **개선 이유:** 변경점을 한 곳에 시각적으로 누적하면 링크 하나로 진척과 근거(전후 캡처·재검증)를 보이고, 거짓 경보로 걸러낸 것도 정직하게 남는다. 판단 시점 규칙은 loop-2에서 실제로 거짓 경보를 낳은 원인을 직접 차단한다.
- **역기능(예측):** 캡처를 data-URI로 박으면 문서 용량이 커진다(항목이 늘수록 수백 KB~MB). 완화: 항목당 핵심 캡처만 담고, 오래된 항목은 캡처를 접거나 요약으로 대체.
- **역기능(실측):** _TBD — 다음 프로젝트 적용 후 갱신._
- **영향 범위:** `SKILL.md`(보고 절에 "루프 변경점 보고 (매 턴, As Is / To Be)" 추가).

## v0.1.1-unproven — 준비 상태 보드·강등 규칙·순차 실행 확정 (2026-07-13)

- **As-Is:** "밖의 외부 조건(코드·인증·흐름 문서·등록부)이 프로젝트마다 있을 수도 없을 수도 있다"는 문제의 통제 규칙이 문서 여기저기 흩어져 있고 암묵적이었다. 오케스트레이터의 임무 흐름에는 0단계(부트스트랩)와 1단계(분모 생성)가 빠져 A단계(페르소나 띄우기)부터 시작했고, 분모를 코드에서 어떻게 만드는지와 인스턴스를 병렬/순차 중 무엇으로 도는지가 미정이었다.
- **To-Be:** 세 가지를 확정했다. (1) 전제조건을 신호등(🟢 충족 / 🟡 보완 / 🔴 차단)으로 평가하는 **"준비 상태 보드"** 를 도입했다. 🔴가 되는 것은 '구동 앱' 하나뿐이고, 나머지는 스킬이 초안을 생성하거나 등급을 강등해 흡수한다. 부트스트랩이 매 실행 이 보드를 평가해 원장 맨 앞에 출력한다. (2) 분모 생성을 **"열거 스크립트=완성형(🟢) + 오케스트레이터 판단=폴백(🟡 저신뢰)"** 등급으로 확정했다. (3) 인스턴스 실행은 **순차를 기본**으로 확정했다(병렬·격리는 후속).
- **문제:** 범용 스킬인데 작동 유무가 목록 밖 외부 변수에 좌우되고, 프로젝트가 체계화되어 있지 않아 밖이 일관되지 않으면 통제할 방법이 없었다.
- **개선 이유:** "외부를 일관되게 갖추라고 요구"하는 대신 "감지 → 등급 강등/초안 생성 → 원장에 명시"로 바꾸면, 밖이 달라도 스킬이 스스로 등급을 낮춰 돌고 그 사실을 정직하게 드러낸다. 완전성 주장의 근거인 분모는 기계 열거(스크립트)를 완성형으로 두어 견고하게 만든다.
- **역기능(예측):** (1) 순차 실행은 느리다 — 규모가 큰 프로젝트에서 시간이 오래 걸릴 수 있다. 완화: loop-1 실측 후 격리 전제 하에 병렬 도입. (2) 신호등 자동 감지가 프레임워크별로 부정확할 수 있다. 완화: 애매하면 보수적으로 낮은 등급(🟡)을 매기고 원장에 근거를 남긴다.
- **역기능(실측):** _TBD — loop-1 후 갱신._
- **영향 범위:** `SKILL.md`(0·1단계), `agents/orchestrator.md`(0·1단계 추가·⚠️ 해소·순차 규칙), `qa-contract.md`(준비 상태 보드 표), `README.md`(관계도·강등 표).

## v0.1.0-unproven — 하네스 최초 정의 (2026-07-10)

- **As-Is:** QA(품질 검증) 하네스가 없었다. 품질 검증은 단일 모델에게 한 번 물어보고, 100가지 가능한 문제 중 "가장 눈에 띄는 5가지"만 뽑아내는(distill, 요약·증류) 방식이었다. 그 결과 나머지 95가지, 즉 드문 경계 상황(엣지 케이스), 진행이 막히는 지점, 여러 사용자가 동시에 쓸 때 생기는 마찰을 잡지 못했다. 테스트 케이스(TC)는 사람이 손으로 작성해서, 이미 아는 내용을 다시 정리하는 수준에 그쳤다.

- **To-Be:** `qa-swarm` 스킬을 도입한다. 검증 대상 전체(분모)를 사람의 기억이 아니라 **코드에서 자동으로 뽑아 커버리지 맵을 만든다.** 이 맵은 라우트(화면 경로), 역할, 액션(동작), 상태를 곱해 만든 조합이며, 사람이 그린 지도가 이미 있으면 그 둘을 합친다(합집합). 그 위에서 다음 주체들이 협력해 검증한다.
  - **정보를 굶긴 페르소나 subagent** — 배경지식을 일부러 제한해, 아무것도 모르는 사용자가 처음 쓰는 상황을 순진하게 재현하는 하위 에이전트. 지형(전체 그림)을 훑는 1차 검증이라 L1이라 부른다.
  - **적대적 전문가** — 일부러 시스템을 깨뜨리려 파고드는 전문가 역할. 깊이 파는 2차 검증이라 L2라 부른다.
  - **오케스트레이터** — 발견 사항을 검증하고 분류하고 취합하는 지휘 주체.
  - **2차원 커버리지 원장** — 무엇을 얼마나 검증했는지를 격자(예: 화면 × 역할)로 기록하는 장부.

  검증에서 나온 발견은 `tc-registry`(테스트 케이스 등록소)에 정식 테스트 케이스로 결정화(고정 등록)한다. 이 모든 구성물은 `.claude/skills/qa-swarm/` 한 폴더 안에 두고, 폴더 밖에는 CI(지속적 통합 자동 검사)용 yml 파일 하나만 둔다.

- **문제:** 완전성(빠짐없이 전부 검증한 상태)을 한 번의 처리(단일 패스)로 "추출"하려다, 매번 꼬리(tail — 드물지만 실제로 존재하는 나머지 경우들)를 놓쳤다.

- **개선 이유:** 완전성은 한 번에 추출되는 것이 아니라 탐색을 반복해서 **제조되는** 것이다. 구체적으로는 여러 관점을 동원하고, 정보를 일부러 제한해(정보 굶김) 순진한 사용자를 재현하며, 적대적으로 파고들고, 이 과정을 반복한다. 아직 검증하지 못한 미지의 영역을 원장의 빈칸으로 드러내(가시화), 목표 자체를 "다 안다"에서 "미지의 영역을 줄이고 그 경계를 파악한다"로 바꾼다.

- **역기능(예측):** 이 변경이 새로 만들 수 있는 부작용을 도입 시점에 다음과 같이 예측했다.
  1. 합성 에이전트(사람을 흉내 낸 자동 검증 주체)는 실제 사람보다 UI를 능숙하게 다룬다. 그래서 구조나 논리가 막히는 문제(블로커)에는 강하지만, 사람이 느끼는 감성적 불편은 실제보다 적게 보고할 수 있다. 완화책: "사람의 실제 사용성 점검이 필요하다"고 판단되는 후보를 별도로 출력한다.
  2. 검증되지 않은 발견 사항(finding)이 그대로 쌓이면 소음이 될 수 있다. 완화책: 오케스트레이터가 각 발견을 재현하고 증거를 남기도록 필수화한다.
  3. 환경 계약(테스트에 필요한 초기 데이터 seed와 인증 auth를 갖추기로 한 약속)을 충족하지 못한 프로젝트에서는 L2 심층 검증과 여러 역할 기반 검증이 절반밖에 작동하지 못한다. 완화책: 그 제약을 원장에 명시하고, 최소한 L1 지형 파악만이라도 가동한다.
  4. 여러 에이전트가 떼로 움직이는 스웜(swarm)이 파일을 무분별하게 만들어낼 위험이 있다. 완화책: 기존 내용을 지우지 않고 덧붙이기만 하는(append-only) 단일 저장소를 쓰고, 파일이 끝없이 늘어나는 것을 막는(반발산) CI 게이트를 둔다.

- **역기능(실측):** 아직 관측되지 않았다. 이 하네스를 처음 도입한 프로젝트에서 첫 검증 사이클(loop-1)을 실행한 뒤에 갱신한다.
