# qa-swarm — 문서 지도와 의존 관계

> 한 줄 요지: 이 파일은 qa-swarm 스킬이 **어떤 문서로 이뤄지고, 각 문서가 무엇을 요구하며, 프로젝트 밖의 조건이 없을 때 어떻게 등급을 낮춰 도는지**를 보여주는 지도다. 스킬의 방법 자체(파이프라인·원리)는 [`SKILL.md`](./SKILL.md)에 있다.

이 지도가 필요한 이유는, qa-swarm이 **범용 스킬**이면서도 실제 작동은 프로젝트마다 다른 외부 조건(코드, 로그인, 흐름 문서, 등록부 등)에 기대기 때문이다. 그 외부 조건은 프로젝트마다 있을 수도 없을 수도 있다. 이 지도는 "무엇이 없으면 무엇을 못 하거나, 대신 무엇을 하는가"를 명시해, 밖이 일관되지 않아도 스킬이 **스스로 등급을 낮춰 돌고 그 사실을 정직하게 드러내도록** 통제한다.

## 문서 의존 관계도

화살표는 "A가 동작하려면 B를 요구한다"를 뜻한다. 색으로 세 종류를 구분한다.

- **파랑** — qa-swarm 스킬 자신의 문서(이 폴더 안).
- **노랑 점선** — 프로젝트가 공급하는 외부 입력. 노드 안에 "없으면 → …" 강등 동작을 적었다.
- **초록** — 스킬이 실행 중에 **스스로 생성**하는 산출물(미리 있을 필요 없음).

색은 상자와 폴백을 나눠 읽는다. **상자 색 = authored 버전을 누가 소유하나.** 그래서 `user-flows.md`처럼 프로젝트가 손으로 쓰는 문서는 상자가 노랑이다. 다만 그 노드의 "없으면 → 스킬이 초안 생성"은 스킬-생성 동작이므로 폴백 문구를 초록으로 적는다. 순수 초록 상자(`tc-registry.md`·2D 원장)는 프로젝트가 절대 authored 하지 않고 스킬만 만드는 것이다. 나머지 "없으면"(코드베이스 저신뢰, 등록부 no-op, e2e 승격 보류, 구동 앱 실행 불가)은 생성이 아니라 강등이다.

## 밖이 없을 때 — 등급과 강등 (통제 규칙)

외부 조건은 전부 같은 "요구"가 아니다. 등급을 나누고, 없을 때의 동작을 미리 정해 둔다. **진짜 필수는 딱 하나 — 구동 가능한 앱 — 이고, 나머지는 감지해서 등급을 낮추고 원장에 명시한다.**

| 외부 변수 | 등급 | 없을 때 동작 |
|---|---|---|
| 구동 앱 + 브라우저 조작(webapp-testing) | 필수(유일한 게이트) | 실행을 거부하고 "무엇이 없어 못 도는지"만 보고한다 |
| 코드베이스(분모 원천) | 유도 | 코드에서 분모를 생성한다. 미지원 프레임워크면 "저신뢰 분모"로 표시한다 |
| 인증(역할별 로그인) | 선택·강화 | L1 도달성만 돌고, L2·다역할은 원장에 빈칸으로 명시한다 |
| 페르소나 문서(user-flows류) | 선택·강화 | 있으면 합집합, 없으면 스킬이 코드에서 초안을 만든다 |
| 파일 등록부(doc-governance 1절) | 선택 | 반발산 게이트를 no-op(경고만)으로 돌린다 |
| e2e 스펙 | 후속 | 승격을 보류하고 결정화 항목만 기록한다 |
| tc-registry 인스턴스 · 2D 원장 | 자가 생성 | 스킬이 만든다(미리 있을 필요 없음) |

## 이 지도는 살아있는 문서다

qa-swarm의 문서 구성이나 외부 의존이 바뀌면, **이 관계도와 강등 표를 같은 변경에서 함께 갱신한다.** 변경 자체는 [`CHANGES.md`](./CHANGES.md)에 As-Is/To-Be 형식으로 기록한다. 지도가 실제 구조와 어긋나기 시작하면, 이 스킬이 "무엇에 기대는지"를 아무도 신뢰할 수 없게 되므로, 갱신을 미루지 않는다.
