---
name: qa-swarm
description: Stakeholder-simulation QA orchestration. Spawns independent stakeholder sub-agents that each live their real jobs in the target system until they succeed, plus adversarial agents that attack success paths — looping until no new findings emerge. Produces a coverage map (state × role × action grid) and a residual-unknown ledger, never a truncated "top N" test case list. Use when the user asks for QA, 품질 검증, 품질 보증, 테스트 케이스 도출, TC 뽑기, 엣지케이스 발견, 커버리지 분석, or comprehensive quality assurance of a feature or system. Do NOT use for running a single quick test (use webapp-testing directly).

# QA Swarm — 이해관계자 시뮬레이션 기반 품질 탐색

이 스킬을 읽고 있는 **당신(메인 세션)이 오케스트레이터**입니다. 당신은 테스트 케이스를
직접 생성하지 않습니다. 당신의 일은 (1) 시스템을 모델링하고, (2) 이해관계자·적대자
서브에이전트를 띄우고, (3) 결과를 격자에 병합하고, (4) 마를 때까지 반복시키고,
(5) 커버리지 맵과 잔여 미지(residual unknown)를 보고하는 것입니다.

## 왜 이 방식인가 (설계 철학 — 반드시 이해하고 시작)

단일 모델에게 "테스트 케이스를 뽑아줘"라고 한 번 요청하면 **분포의 머리(head)**만
나옵니다. 현저성(salience) 높은 중요 케이스 N개. 하지만 QA의 본질은 꼬리(tail) —
엣지케이스와 놓친 부분 — 이고, 꼬리는 정의상 현저성이 낮아 단일 패스로는 안 떠오릅니다.

**완전성은 추출(extract)이 아니라 탐색으로 제조(manufacture)하는 것입니다.**
탐색에는 네 가지 속성이 필요하며, 이 스킬의 모든 단계는 이 넷을 강제하기 위해 존재합니다:

1. **열거 모드** — "중요한 것만"이 아니라 지루하고 뻔한 것까지 전부. 탐색 단계에서
   랭킹·요약·절삭(truncation) **금지**. 우선순위는 마지막 보고서에서만 매깁니다.
2. **독립된 다관점** — 이해관계자 에이전트들은 서로의 결과를 **모르는 채(blind)** 생성합니다.
   관점이 섞이면 겹치는 영역만 나옵니다.
3. **적대적 탐색** — "사용자가 뭘 하나"(서술)가 아니라 "이걸 어떻게 깨뜨리나"(공격).
   엣지케이스는 흐름을 설명할 때가 아니라 공격할 때 나옵니다.
4. **마를 때까지 반복** — 연속 2라운드에서 새 발견이 0이 될 때까지. 한 번 돌린 결과로
   끝내면 항상 꼬리가 남습니다.

그리고 핵심 인식 하나: **놓친 케이스의 대부분은 창의적 발명이 아니라 아무도 끝까지
채우지 않은 격자의 빈 칸**입니다. 상태 × 역할 × 액션을 곱하면 수백 칸이 나오고,
"InApproval 상태에서 viewer가 URL 직접 치면?"은 발명이 아니라 격자 채우기입니다.

최종 산출물은 "중요 TC 22개" 리스트가 **아닙니다**. 산출물은
**커버리지 맵**("격자 480칸 중 310 확인 · 120 미검 · 50 설계상 불가")과
**잔여 미지 장부**(아직 안 본 곳이 어디인지 명시)입니다. 미지를 숨기지 말고 보이게 만드세요.

## 동작 모드 결정 (가장 먼저)

| 모드 | 조건 | 에이전트가 하는 일 |
|------|------|------------------|
| **LIVE** | 실행 중인 앱 URL이 있거나 로컬에서 띄울 수 있음 | `webapp-testing` 스킬로 실제 앱을 Playwright로 조작 — 진짜로 "써본다" |
| **SPEC** | 코드/스펙 문서만 있음 | 코드·스펙을 근거로 경로를 정밀하게 걸어봄(walk-through) — 모든 주장에 파일:라인 근거 필수 |

가능하면 항상 LIVE를 선택하세요. 페르소나가 실제로 부딪혀야 낮은 현저성 케이스가
나옵니다. LIVE가 가능한지 먼저 확인(dev 서버 기동 시도)하고, 안 되면 SPEC으로
진행하되 보고서에 "SPEC 모드 한계"를 명시합니다.

## 작업 디렉토리

세션 스크래치패드(또는 사용자가 지정한 곳)에 `qa-swarm/` 폴더를 만들고 모든 중간
산출물을 파일로 남깁니다. 에이전트 간 전달은 파일 경로로 합니다 (컨텍스트 절약).

```
qa-swarm/
  system-model.md      # Phase 0 산출물
  stakeholders.md      # Phase 1 산출물
  grid.md              # Phase 2 격자 (라운드마다 갱신)
  rounds/round-N/      # 라운드별 에이전트 원본 보고서
  findings.md          # 중복 제거된 발견 누적 (seen 전체, 기각된 것 포함)
  report.md            # 최종 보고서
```

## Phase 0 — 시스템 모델링 (오케스트레이터 직접 수행)

스펙·코드·앱을 읽고 `system-model.md`에 추출:

- **객체(objects)**: 시스템이 다루는 엔티티들
- **상태(states)**: 각 객체의 상태 enum과 전이(transition) 규칙
- **역할(roles)**: 권한 체계 — 코드의 permission 체크를 실제로 grep해서 확인
- **액션(actions)**: 각 역할이 수행 가능한 조작 (UI 버튼 + API 엔드포인트 + URL 직접 접근)
- **불변식(invariants)**: "절대 일어나면 안 되는 것" — 이게 적대 라운드의 표적

## Phase 1 — 이해관계자 전수 조사 (오케스트레이터 직접 수행)

`stakeholders.md`에 **명백한 역할 너머까지** 열거:

- 1차 역할들 (구매담당, 승인자, 품질, viewer 등 도메인 역할)
- 관리자 / 시스템 설정자
- **갓 온보딩된 신규 사용자** (아무것도 모르는 상태)
- 파워 유저 (대량 작업, 단축 경로)
- **실수를 수습하는 사람** (잘못 올린 것 취소·롤백하려는 사람)
- 권한이 없는/애매한 사람 (링크만 받은 사람, 세션 만료된 사람)
- API/연동 소비자 (있다면)
- 동시에 같은 객체를 만지는 두 사람 (이건 페르소나가 아니라 시나리오지만 여기서 등록)

각 이해관계자에 **jobs-to-be-done**(그 사람이 시스템으로 끝내야 하는 실제 업무 목록)을 적습니다.

## Phase 2 — 격자 열거 (오케스트레이터 직접 수행, 기계적으로)

`grid.md`에 **상태 × 역할 × 액션** 매트릭스를 만듭니다. 수백 칸이 나와도 전부 만듭니다
— 이 단계가 지루하면 제대로 하고 있는 것입니다. 각 칸의 초기 상태는 `⬜ 미검`.

칸 상태 코드: `✅ 확인됨(정상)` / `🐞 결함 발견` / `⬜ 미검` / `🚫 설계상 불가` / `❓ 스펙 모호`

`🚫`는 반드시 근거(코드 위치나 스펙 조항)와 함께. 근거 없는 🚫는 ⬜입니다.

## Phase 3 — 이해관계자 팬아웃 (병렬, blind)

이해관계자마다 `qa-stakeholder` 에이전트를 **하나의 메시지에서 병렬로** 띄웁니다
(Workflow 툴이 있으면 그걸로, 없으면 Agent 툴 다중 호출). 각 에이전트에게 전달:

- 자신의 페르소나와 jobs-to-be-done (그 사람 것만 — 다른 페르소나 정보 주지 않기)
- `system-model.md` 경로, 동작 모드(LIVE면 URL/기동 방법), 작업 폴더 경로
- **골(goal)**: "당신은 X입니다. 이 시스템에서 당신의 실제 업무를 **끝까지 완수하세요.
  성공할 때까지.** 막힌 곳, 헤맨 곳, 애매했던 곳, 우회한 곳, 놀란 곳을 **전부** 기록하세요.
  중요해 보이는 것만 고르지 말고 사소한 것까지 전부."
- 보고서를 `rounds/round-N/<페르소나>.md`에 쓰게 함 (형식은 에이전트 정의에 있음)

핵심: 이들은 테스터가 아니라 **업무를 끝내려는 당사자**입니다. "쓸 수 있을 때까지
써본다"는 골이 중요한 이유 — 케이스를 *떠올리는* 게 아니라 *부딪히게* 만들어서
현저성 낮은 문제가 자연히 드러납니다.

## Phase 4 — 적대 라운드 (병렬)

Phase 3 보고서가 모이면 `qa-adversary` 에이전트들을 띄웁니다. 각 적대자에게
**서로 다른 렌즈** 하나씩 배정 (동일 프롬프트 N개 금지 — 다양성이 중복보다 많이 잡습니다):

- **경계값**: 0, 1, 최대, 최대+1, 빈 문자열, 초장문, 특수문자/이모지/RTL
- **불법 상태 조합**: 격자에서 ⬜/❓인 칸 중 불변식을 위협하는 것들 직접 타격
- **동시성**: 두 사용자가 같은 객체를 동시에 — 승인 직전 취소, 수정 중 삭제
- **권한 우회**: URL 직접 접근, API 직접 호출, 세션 만료 후 재사용, 뒤로가기
- **중단·복구**: 진행 중 이탈, 네트워크 끊김, 새로고침, 중복 제출
- **교차 심문**: 이해관계자 A의 성공 경로를 B의 입장에서 공격 — "A의 이 성공이
  B에게 문제를 만드는가?" (승인자의 빠른 승인이 품질담당에게는 검토 기회 상실일 수 있음)

각 적대자에게 관련 이해관계자 보고서와 `grid.md`를 전달하고, 재현 절차와 함께 보고하게 합니다.

## Phase 5 — 병합 (오케스트레이터 직접 수행)

- 라운드 보고서들을 읽고 `findings.md`에 병합. **중복 제거는 seen 전체 기준**
  (확정된 것 기준이 아님 — 기각된 발견이 매 라운드 되살아나면 수렴하지 않습니다).
- 발견마다 격자의 해당 칸 상태를 갱신. 에이전트가 "확인했다"고 한 칸은 ✅로.
- 의심스러운 발견(재현 절차가 없거나 근거가 약한 것)은 다음 라운드에 검증 태스크로 넘김.

## Phase 6 — 마를 때까지 반복

`qa-auditor` 에이전트를 하나 띄워 묻습니다: "이 격자와 발견 목록에서 **빠진 것**은
무엇인가 — 안 돌린 렌즈, 안 세운 페르소나, 미검인 채 방치된 칸 무더기, 근거 없는 🚫?"

auditor의 답이 다음 라운드의 작업 목록이 됩니다. Phase 3–5를 반복하되:

- **종료 조건**: 연속 2라운드에서 (새 발견 0) AND (새로 채워진 칸 0)
- 라운드마다 `log`처럼 사용자에게 진행 상황 보고: "라운드 3: 새 발견 4건, 격자 310/480"
- 토큰 예산이 걸려 있으면 라운드 수를 예산에 맞춰 조절하되, **조기 종료했다면
  보고서에 어디까지 돌았는지 명시** (조용한 절삭 금지)

## Phase 7 — 최종 보고서 (`report.md`)

순서 고정:

1. **커버리지 맵** — 격자 요약 수치와 시각화 (칸 수 기준: ✅/🐞/⬜/🚫/❓).
   이것이 대표 산출물입니다. "몇 개 찾았다"보다 "어디를 봤고 어디를 안 봤다"가 먼저.
2. **발견 목록** — 이제서야 심각도로 정렬. 각 발견에 재현 절차, 발견한 페르소나/렌즈, 근거.
3. **잔여 미지 장부** — ⬜와 ❓로 남은 것들, 왜 못 봤는지(모드 한계·예산·환경 부재),
   사람이 직접 봐야 할 것 추천.
4. **스펙 모호 목록** — ❓ 칸들은 버그가 아니라 의사결정 대기 항목. 따로 분리.

사용자가 원하면 HTML 아티팩트로 렌더링하되, HTML은 그릇일 뿐임을 기억하세요 —
내용의 상한은 탐색이 결정합니다.

## 금지 사항 (요약)

- 탐색 단계에서 "중요한 것만" 추리기 — 절삭은 완전성의 적
- 페르소나 성공 = 검증 완료로 간주 — 성공은 happy path 하나가 통과했다는 뜻일 뿐
- 근거 없는 🚫(설계상 불가) 판정
- 발견 개수를 미리 정해두기 ("TC 20개 뽑기") — 개수는 탐색이 결정
- 한 라운드로 종료 — 최소 auditor 1회 + 재라운드 1회는 돌 것
