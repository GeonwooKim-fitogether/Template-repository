# 환경 계약 (QA Contract)

> **한 줄 요지:** 이 문서는 QA 스웜(여러 가상 사용자 subagent가 실제로 도는 앱을 직접 조작하며 문제를 찾는 품질 검증 무리)이 **프로젝트의 앱을 실제로 구동하기 위해 필요한 입력**을 정의하는 계약이다. 그 입력의 **대부분은 하네스가 자동으로 탐지**하고, 사람은 **자동으로 알아낼 수 없는 인증(역할별 로그인) 하나만** 공급한다.

## 이 계약은 무엇이고, 왜 필요한가

QA 스웜을 돌리려면 먼저 "이 프로젝트의 앱을 어디서, 어떻게 띄우고, 어떤 데이터로, 누구로 로그인해 조작하는가"가 정해져 있어야 한다. 이 문서가 그 정보를 담는 계약이다.

**이 계약은 사람이 백지에서 손으로 채우는 양식이 아니다.** 이유는 두 가지다.

첫째, 여기 담기는 사실 대부분은 이미 프로젝트 코드와 설정 안에 있다. 하네스(이 스킬을 돌리는 자동화 틀, orchestrator)가 파이프라인의 **0단계 부트스트랩(코드를 훑어 필요한 입력의 초안을 스스로 만들어 두는 단계)** 에서 이 값들을 자동으로 탐지해 채운다. 사람이 할 일은 그 **탐지 결과가 맞는지 확인**하는 것이다.

둘째, 손으로 유지하는 사실은 시간이 지나면 실제 설정과 어긋난다(드리프트한다). 그래서 이 계약은 한 번 적어 두고 방치하는 것이 아니라 **매 실행마다 다시 탐지하는 것**을 원칙으로 한다.

특정 프로젝트의 실제 값(실제 주소, 실제 시드 파일 이름 등)을 찍어 둔 스냅샷은, 그 프로젝트에서 부트스트랩이 생성한다. 이 템플릿 파일에는 두지 않는다. 아래 표의 값들은 모두 **스택(기술 구성)과 무관한 예시**이며, 실제 프로젝트가 Next.js·Supabase든 Rails든 Django든 탐지하는 대상만 달라질 뿐 원리는 같다.

## 무엇을 어떻게 채우나 — 두 갈래

계약을 채우는 방법은 두 갈래로 나뉜다. 하나는 하네스가 자동으로 채우는 항목이고, 다른 하나는 사람이 직접 공급하는 유일한 항목이다.

### 갈래 1 — 하네스가 자동 탐지한다 (매 실행마다 다시 채움)

하네스가 프로젝트의 설정 파일과 코드를 훑어 다음 다섯 항목의 초안을 만든다. 각 항목이 **무엇을 뜻하는지**와, 하네스가 그것을 **어디서 찾아내는지**는 다음과 같다.

| 채울 항목 | 뜻 (처음 보는 사람을 위한 설명) | 하네스가 찾는 곳 (예시) |
|---|---|---|
| baseURL 및 서버 기동법 | 앱의 접속 주소(baseURL)와, 그 앱을 어떤 명령으로 띄우는가 | 테스트 러너 설정. 예: `playwright.config`의 `webServer` 항목과 포트 번호 |
| 픽스처 모드 vs 실DB 모드 | 검증에 미리 준비한 가짜 데이터(픽스처, fixture)를 쓸지, 실제 데이터베이스에 붙을지 | 데이터베이스 연결용 환경변수(env)가 설정돼 있는지 여부. 예: Supabase의 `NEXT_PUBLIC_*` 변수 |
| 시드 명령 | 검증용 초기 데이터를 앱에 심는(seed) 명령이 있는지, 있다면 무엇인지 | 시드 스크립트의 존재 여부. 예: `supabase/DEMO_SEED*.sql` 파일 |
| 페르소나 및 목표 | 어떤 가상 사용자(페르소나)가 무슨 일을 해내려 시도할지 | 사용자 흐름 문서가 **있으면** 거기서 가져오고, 없으면 코드의 역할·라우트(화면 경로)에서 초안을 뽑는다 |
| 산출물 위치 | 발견과 원장을 어느 파일에 기록할지 | 관례에 따른 기본값. 예: `tc-registry`(테스트 케이스 등록부) 대장 파일 |

### 갈래 2 — 사람이 공급하는 유일한 것: 인증

위 다섯 항목과 달리, **인증(역할별 로그인 정보)은 자동으로 추론할 수 없고 비밀에 해당하므로 자동화되지 않는다.** 이것 하나만 사람이 직접 공급한다.

이때도 인증을 **글로 적어 두지 않고, 실제로 실행되는 것으로 둔다.** 구체적으로는 다음과 같다.

- **역할별 로그인**은 테스트 러너의 **setup 스크립트**로 둔다. 이 스크립트는 로그인한 상태를 저장한 파일(예: Playwright의 `storageState` — 로그인 세션을 담아 재사용하는 저장 상태)을 만들어 두고, 하네스가 그것을 가져다 쓴다.
- 어떤 역할의 로그인이 **제공되지 않으면**, 그 역할에 대한 L2(Layer 2, 정상 흐름을 깊게 공격해 깨뜨리는 적대적 검증)와 권한 검증은 건너뛴다. 대신 로그인 없이도 확인할 수 있는 범위, 즉 **L1(Layer 1, 처음 쓰는 사용자가 목표에 도달할 수 있는지를 넓게 훑는 검증)의 도달성까지만** 돌리고, 그 제약을 원장(검증 결과를 기록하는 표)에 명시한다.

## 준비 상태 보드 (신호등)

부트스트랩(0단계)이 매 실행 첫머리에 아래 전제조건을 하나씩 자동 감지해 신호등을 매기고, 이 보드를 원장 맨 앞에 출력한다. 신호등 규칙은 이렇다. **🟢 충족**(그 기능 완전 가동) / **🟡 보완**(스킬이 초안을 생성하거나 등급을 강등해 계속 실행) / **🔴 차단**(실행 불가). **🔴가 될 수 있는 것은 "구동 앱" 하나뿐**이고, 나머지는 없어도 아래 "없으면 대응"대로 흡수된다.

| 전제조건 | 없으면 대응 | 🟢(초록) 판정 기준 |
|---|---|---|
| 구동 앱 + 브라우저 조작 | 차단 — 실행 거부, 부족분만 보고 | baseURL·기동법 탐지 + webapp-testing 사용 가능 |
| 코드에서 분모 생성 | 스킬 생성 — 없으면 판단으로(저신뢰) | 라우트·역할·상태 열거 스크립트 존재 |
| 역할별 인증 | 등급 강등 — L1 도달성만 | 역할별 로그인 setup 스크립트 존재 |
| 페르소나·목표 | 스킬 생성 — 코드서 초안 | 흐름 문서(user-flows류) 존재 |
| 시드 데이터·격리 | 등급 강등 — 상세 흐름 진입 제한 | 시드 리셋 명령 + 전용 워크스페이스 격리 |
| 파일 등록부 | 등급 강등 — 반발산 게이트 no-op(경고만) | 등록부(예: doc-governance 1절) 존재 |
| e2e 승격 대상 | 등급 강등 — 승격 보류, 결정화만 기록 | e2e 스펙 경로 존재 |

이 보드가 곧 "밖이 프로젝트마다 달라도 스킬이 스스로 등급을 매기고 그 사실을 드러내는" 통제 장치다. 각 항목의 자세한 탐지 위치는 위 "갈래 1" 표에 있다.

**"코드에서 분모 생성" 항목은 완료 게이트다.** 다른 항목과 달리, 이 항목이 🟢/🟡로 채워지기 전에는 **L1 커버리지를 "충분/완료"로 선언할 수 없다.** 분모(다 봐야 할 목록)가 없으면 "얼마나 봤나"를 잴 기준 자체가 없어, 페르소나 몇 개만 돌리고 완료로 착각하기 쉽기 때문이다. 분모가 없으면 L1은 "측정 불가(분모 미생성)"로 보고한다(SKILL.md 보고 절의 완료 게이트 참조). 이 규칙은 첫 실전에서 분모를 건너뛴 채 대표 여정 몇 개만 돌리고 "L1 마무리"로 오판한 실제 사고를 막기 위한 것이다.

## 정리

결국 사람이 개입하는 몫은 "계약서를 작성하는 것"이 아니라, **인증 셋업 스크립트 1개를 준비하고, 하네스가 자동 탐지한 결과를 확인하는 것** 두 가지뿐이다. 나머지는 프로젝트 설정에서 매 실행 다시 탐지되므로 낡지 않는다. 프로젝트의 기술 스택이 무엇이든 이 원리는 동일하며, 스택마다 달라지는 것은 하네스가 탐지하는 대상뿐이다.
