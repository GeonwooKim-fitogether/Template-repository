# TC 대장 (Test Case Registry) — 포맷 템플릿

> **한 줄 요지.** 이 문서는 각 사용자 흐름을 **합격/불합격을 명확히 가를 수 있는 최소 검증 단위(TC, Test Case = 테스트 케이스)** 로 쪼개어, 그 단위마다 "구현이 되어 있는가, 무엇을 충족해야 합격인가, 마지막 검증은 어땠는가"를 한 표에 모아 두는 대장(臺帳, 기록 장부)이다.

## 이 대장이 하는 일과, 다른 산출물과의 분업

QA(품질 보증) 작업에는 두 종류의 산출물이 있고, 이 대장은 그중 하나다. 둘의 역할은 다음과 같이 나뉜다.

- **지도 (서술형 워크스루, 예: `test-cases.html`).** 흐름이 "어떻게 진행되는지"를 이야기처럼 풀어 설명하는 문서다.
- **대장 (이 문서).** 흐름을 판정 가능한 단위로 쪼개어 "합격이냐 불합격이냐"를 기록하는 표다.

즉, **서술은 지도가 맡고, 판정은 이 대장이 맡는다.** QA 스웜(swarm = 떼. 여러 QA 에이전트를 페르소나로 띄워 실제 앱을 직접 조작하며 문제를 찾는 방식)이 확인한 발견은 이 대장의 행으로 결정화되어 남는다. 그리고 e2e 테스트(end-to-end 테스트 = 실제 사용자 흐름을 처음부터 끝까지 자동으로 실행해 검증하는 테스트) 스펙이 각 TC-ID를 주석으로 달아 이 대장과 코드를 연결한다.

이 파일 자체는 **포맷 템플릿**이다. 실제 프로젝트는 이 구조를 그대로 가져다 자기 대장을 만든다.

## 판정 규칙 (공통)

두 가지를 각각 다른 기호로 표시한다. 하나는 **구현 상태**(코드가 실제로 있는가)이고, 다른 하나는 **검증 결과**(돌려 봤더니 합격인가)이다.

**구현 상태 — 코드가 실행 대상으로 존재하는가**

| 기호 | 뜻 |
|---|---|
| ✅ | 실행에 필요한 액션·화면·데이터가 실제로 존재한다 (검증 대상이 된다) |
| 🟡 | 부분 구현 — 되는 범위까지만 존재한다 |
| ❌ | 미구현 — 아직 실행하지 않는다 (실패가 아니라 대상이 아직 없는 상태) |
| ⏸️ | 의도적으로 보류한 상태 |

**검증 결과 — 실제로 돌려 본 마지막 판정**

| 기호 | 뜻 |
|---|---|
| P | Pass. 판정기준을 충족했다 (검증 일자를 함께 적는다) |
| F | Fail. 판정기준을 위배했다 (관련 이슈를 함께 적는다) |
| ⚠️ | 동작은 하지만 우려가 있다 (도메인 불변식이나 사용자 경험(UX) 측면의 문제) |
| – | 아직 실행하지 않았다 |

TC-ID는 `TC-<모듈약자>-<번호>` 형식으로 붙인다. 모듈 약자는 각 프로젝트가 정의한다.

## 행 포맷

대장의 각 행은 다음 아홉 개 칸으로 이루어진다.

| TC | 흐름# | 이름 | 행위자 | 전제(요약) | 판정기준 (합/불 한 줄) | 구현 | e2e | 최근 검증 |
|---|---|---|---|---|---|---|---|---|
| TC-XXX-01 | (지도 WF#) | | | | | ✅ / 🟡 / ❌ / ⏸️ | 스펙명 | P / F / ⚠️ / – + 일자 |

여기서 "흐름#"의 WF는 지도 문서 안에서 각 흐름(Work Flow)에 매긴 일련번호를 가리킨다.

## 이 대장은 손으로 쓰는 것이 아니라 생성된다 (지도가 없어도 된다)

대장의 각 행은 두 가지 재료에서 자동으로 생성된다. 첫째는 **코드**로, 라우트(화면 주소)·액션·역할·상태는 코드에 항상 존재한다. 둘째는 **QA 스웜이 실제로 발견한 내용**이다.

지도(`test-cases.html`)나 user-flows 문서는 프로젝트에 있을 수도 있고 없을 수도 있다. **있으면** 교차 참조해 완전성을 보강한다(즉 "문서로 정의된 흐름까지 빠짐없이 커버했는가"를 확인한다). **없어도 무방하다.** 스웜이 도메인 목표를 기준으로 흐름을 스스로 발견하기 때문이다.

각 칸이 어디서 채워지는지는 다음과 같다. 아이콘의 뜻은 🤖 = 자동 생성, 🧑 = 사람이 직접 판단해 채움이다.

| 칸 | 무엇을 담나 | 어디서 오나 (우선순위) |
|---|---|---|
| **TC** | 고유 ID (예: `TC-CHG-07`) | 🤖 자동 (순번) |
| **흐름# (출처)** | 이 TC가 어디서 비롯됐는가 | 🤖 지도가 있으면 지도의 WF 번호, 없으면 `스웜:L1` 또는 `코드:라우트`로 표기 |
| **이름** | 사용자 목표를 한 줄로 | 🤖 코드(라우트·액션), 또는 지도가 있으면 지도의 WF 제목 |
| **행위자** | 이 흐름을 수행하는 역할 (예: ENG·APPR·ADMIN·PM·QA·VIEW) | 🤖 코드의 역할·권한(auth) 정보 |
| **전제 (요약)** | 흐름 시작 전에 갖춰져야 할 상태·데이터 | 🤖 코드가 요구하는 선행 상태 + 스웜이 실제로 부딪힌 전제 |
| **판정기준** | **합격과 불합격을 가르는, 관찰 가능한 한 줄** — 이 대장의 심장 | 🤖 초안은 자동(코드 동작 + 지도가 있으면 지도의 '기대값') + 🧑 **사람이 합/불 이분법으로 다듬음** — 여기가 사람이 손대는 유일한 칸이다 |
| **구현** | ✅ 🟡 ❌ ⏸️ — 코드가 있는가 | 🤖 오케스트레이터(스웜을 조율하는 상위 에이전트)가 코드와 대조 |
| **e2e** | 이 흐름을 기계로 강제하는 테스트 스펙의 이름 | 🤖 스펙 파일 검색(grep) |
| **최근 검증** | 마지막 판정(P / F / ⚠️ / –)과 그 일자 | 🤖 스웜이나 e2e 실행 결과가 기록 |

정리하면, 대장은 대부분 자동으로 채워지고 **사람이 판단으로 더하는 부분은 `판정기준` 한 칸뿐**이다.

## 채운 예시 (형식 이해용)

> 아래 행은 **형식을 보여주기 위한 예시**다. 특정 사내 하드웨어팀 사례에서 가져온 것이며, **당신 프로젝트의 데이터가 아니다.** 자기 대장에는 자기 프로젝트의 흐름을 채운다.

| TC | 흐름# | 이름 | 행위자 | 전제(요약) | 판정기준 (합/불 한 줄) | 구현 | e2e | 최근 검증 |
|---|---|---|---|---|---|---|---|---|
| TC-CHG-07 | 47 | Make Effective = 척추 승급 | ADMIN | ECO가 InApproval 상태 + 모든 Stage 승인 완료, 대상 부품은 rev A / InDesign | 승인이 끝나지 않은 상태에서는 버튼이 비활성이고, Effective가 된 뒤에는 부품이 **새 Released rev + Phase 승급**되며 **오직 이 경로로만** 바뀌면 합격(P) | ✅ | changes.spec.ts | ⚠️ 07-10 |

**이 예시 행을 읽는 법.** 행위자인 ADMIN(관리자)이, 전제인 "승인이 끝난 ECO(Engineering Change Order = 설계변경 지시)"에서 Make Effective(변경을 실효화)를 누르면, 부품의 rev(개정 번호)와 phase(수명 단계)가 오직 이 경로로만 바뀌어야 한다는 것이 판정기준이다. 이 판정기준은 이 시스템의 척추(Spine) 규칙 — 부품의 Rev·Phase는 오직 Change의 Effective로만 바뀐다는 핵심 불변식 — 을 그대로 검사한다.

구현은 되어 있고(✅), `changes.spec.ts`라는 e2e 스펙이 이 규칙을 기계로 강제한다. 다만 최근 검증 결과가 ⚠️(우려)인데, 이는 "화면 헤더에는 'Routing 01 CTO'라고 표시되는데 본문에는 '라우팅 없음'으로 나오는 불일치"가 관찰됐기 때문이다. 따라서 이 ⚠️ 항목이 다음 수정 대상이 된다.

## 신규 후보 TC 섹션 (지도에 없던 흐름)

QA 스웜이 발견한, 지도에는 없던 새 사용자 흐름을 여기에 모은다. 특히 L1(스웜의 넓은 지형 탐색 단계 — 시스템 설명을 받지 않은, 처음 쓰는 사용자 페르소나)이 이런 흐름을 자주 찾아낸다. 팀이 합의하면 지도에 정식 WF(흐름)로 승격한 뒤 정식 TC-ID를 부여한다.

| 후보 | 흐름 | 왜 필요한가 | 발견 출처 |
|---|---|---|---|
| X-01 | | | (L1 스웜 / 감사 / …) |

---
*스웜 오케스트레이터는 이 대장에 **행을 덧붙이기만** 한다. 새 파일을 만들지 않는다.*
