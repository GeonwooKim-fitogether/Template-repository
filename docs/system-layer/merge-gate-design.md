# 장치 5 — 머지 게이트 (Merge Gate) 설계

> **한 줄 요지:** 여러 팀이 `main`에서 브랜치를 따 병렬로 일할 때, **오래된 base에서 딴 브랜치가 최신화 없이 머지되어 남의 최신 작업을 조용히 되돌리는(rollback)** 문제를 막는 장치다. 지금까지 대표가 혼자 수동으로 하던 "머지해도 되나"의 통합 판단을, KimOrchestrator(시스템 오케스트레이터)가 자동으로 대신한다.

- **층:** 시스템 오케스트레이터(KimOrchestrator) — KimLead가 아니다.
- **연계:** 장치 3 `pr-gate`(검사 재사용) · 장치 4 계약 레지스트리(충돌 판정 근거) · 장치 2 `sysreport`(결정 큐) · GitHub 가드레일(설정).
- **상태:** 설계 정본. 구현체는 `.claude/commands/merge-gate.md`.

---

## 1. 무엇을 막는가 — 롤백의 기전

"머지했는데 오히려 되돌아간다"는 아무 때나 생기지 않는다. **여러 브랜치가 같은 파일을 건드릴 때** 생긴다.

1. `main`이 A일 때, 브랜치 X·Y를 둘 다 A에서 딴다.
2. Y가 먼저 머지돼 `main`이 B(= A + Y의 수정)가 된다.
3. X는 여전히 옛 A 기준으로, `STATUS.md`·디자인 토큰(`globals.css`)·`sitemap-erd.html`·마이그레이션 같은 **공유 파일**을 건드린 채 머지된다. 그 파일에서 Y의 수정과 충돌한다.
4. 그 충돌을 X의 옛 내용 쪽으로 풀면 **Y의 최신 작업이 조용히 사라진다** — 이것이 롤백이다.

**한 줄 진단: 공유 표면(shared surface) + 오래된 base + 직렬화 안 된 머지, 이 셋이 겹칠 때** 롤백이 난다. 통합을 대표가 혼자 하니 "머지 전에 최신화했나 / 다른 브랜치가 같은 걸 건드리나"를 강제하는 장치가 없어 새는 것이다.

## 2. 두 겹 — 설정이 대부분, 장치는 판단만

롤백의 대부분은 GitHub 설정만으로 사라진다. 장치는 설정이 못 보는 "사람 판단이 필요한 것"만 올린다.

### A. GitHub 가드레일 (설정 · 즉시 · 대부분 해결)

`main` 브랜치 보호에 세 가지를 켠다(코드가 아니라 repo Settings → Branches).

1. **머지 전 브랜치 최신화 강제** (Require branches to be up to date before merging) — 오래된 base 머지 자체를 불가능하게. 각 PR은 머지 직전 최신 `main`을 반영하고 CI를 다시 통과해야 한다.
2. **머지 큐** (Merge queue) — PR을 한 줄로 세워, 각 PR을 "앞 큐까지 반영된 main" 위에서 테스트하고 하나씩 머지. 동시 머지 레이스 제거.
3. **CI(verify) 통과 필수.**

### B. 머지 게이트 (장치 · 판단만 올림)

설정이 못 보는 것 — "이 머지가 남의 것을 지우나 / 같은 계약을 동시에 건드리나".

## 3. 머지 직전 5단계

1. **최신화 확인** — 이 PR이 최신 main 기준인가? behind면 보류(갱신 후 재검증). 가드레일 A가 켜져 있으면 GitHub가 강제하므로 확인만.
2. **pr-gate 재실행** — 척추·문서 갱신 매트릭스·타 파트 영향(장치 3)을 머지 직전 상태로 다시 검사.
3. **계약 충돌 교차 탐지** — 계약 레지스트리(장치 4)로, 지금 열린 다른 PR 중 같은 공유물(파일·표·토큰)을 건드리는 게 있나 대조. 있으면 그게 롤백 위험의 정체. (예: 두 PR이 둘 다 `globals.css` 토큰을 건드림 → 먼저 머지된 쪽이 이기고 뒤가 되돌릴 위험.)
4. **되돌림 사전 시뮬레이션** — 머지 전에 3-way 병합을 dry-run 해, "이 머지가 최근 커밋이 넣은 줄을 지우나?"를 검사. 지운다면 차단·경고.
5. **머지 큐로 하나씩** — 통과분만 큐에 넣어 직렬로 머지.

판정: **통과 → 머지 큐 · 보류 → 최신화·재검증 · 차단 → 결정 큐(사람)**.

## 4. 두 핵심 검사의 구현 방법

| 검사 | 어떻게 |
|---|---|
| **계약 충돌 교차 탐지** | 열린 PR 각각의 변경 파일 목록을 모아 계약 레지스트리의 공유물(파일·표·토큰·흐름)에 매핑. 둘 이상의 열린 PR이 같은 계약에 매핑되면 충돌 후보로 표시하고 소비자 파트와 함께 결정 큐로 올림. |
| **되돌림 사전 시뮬레이션** | 임시 워크트리에서 `git merge --no-commit --no-ff`로 병합을 시늉만 낸 뒤 결과를 base와 diff. 병합 결과가 main의 최근 커밋이 추가한 줄을 삭제(net 제거)하면 되돌림으로 판정해 차단. 순수 추가·비충돌은 통과. |

## 5. 이 검사가 못 잡는 것 (정직하게)

dry-run은 **텍스트 충돌**만 본다. 두 PR이 서로 다른 파일을 고쳤지만 **의미상 충돌**(예: 한쪽이 컬럼을 지우고 다른 쪽이 그 컬럼을 참조)하는 경우는 텍스트로는 안 잡힌다 — 그건 pr-gate의 척추·계약 검사와 e2e가 잡아야 한다. 그래서 게이트는 세 검사를 **겹쳐서** 쓴다.

## 6. 형태 · 전제 · 한계

- **형태:** 커맨드 `/merge-gate <PR>` — 위 5단계를 실행해 판정. GitHub 머지 큐와 연동(가드레일 A). 차단·충돌은 `/sysreport` 결정 큐로.
- **전제:** 가드레일 A는 사람이 설정에서 켜야 함(게이트가 대신 못 켬). 계약 레지스트리가 최신이어야 교차 탐지가 정확.
- **한계:** dry-run은 텍스트 충돌만 — 의미 충돌은 pr-gate·e2e 몫.

## 7. 롤아웃 순서 (효과 대비 비용)

1. **가드레일 A 켜기** — main 보호에 최신화 강제 + 머지 큐 + CI 필수. 설정만, 즉시, 롤백 대부분 제거.
2. **`/merge-gate` 커맨드** — pr-gate 확장 + 계약 충돌 교차 탐지.
3. **되돌림 시뮬레이션 스크립트** — 워크트리 dry-run diff 판정.
4. **머지 큐 연동 + sysreport 결정 큐 배선.**

## 8. KimLead와의 구분 (재확인)

머지 게이트는 **KimOrchestrator(시스템 오케스트레이터) 층**의 장치다. KimLead(세로 — 한 팀이 한 요청을 잘 만들게 지휘)와 달리, 이 층은 가로 — 여러 팀이 병렬로 일해도 전체가 안 어긋나게 — 를 맡고, 머지 게이트는 그 "통합 권한"을 구현한다. 정의는 `.claude/commands/kimorchestrator.md` 참조.
